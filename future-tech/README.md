# future-tech/ — 未来技术调研层

从每日新闻里**收藏**一条值得深挖的新闻（多为科技/未来技术类），以它为起点写一篇
**结构化深度调研报告**，归档到这里。和 `digests/` 一样，**markdown 是唯一事实来源**，
`web/build.py` 只做渲染。

## 两个组成部分

1. **待办清单（watchlist）— 存在 GitHub Gist，不入库。**
   站点里每条新闻旁有 ☆ 收藏按钮，点击把该条（标题 + 链接 + 时间）写进你的一个
   私密 Gist。未来技术页顶部会读取这个 Gist 渲染成"待调研"清单。Gist 的 id 和
   token 存在浏览器 localStorage，**绝不进仓库**（见站点「未来技术」页的 ⚙ 设置）。

   > 为什么用 Gist：站点是本地 `file://` 静态页，没有后端，按钮无法直接写文件/git。
   > Gist + GitHub API 支持跨域读写，是纯前端能落地的最轻方案。

2. **报告（reports）— 就是本目录下的 markdown，入库。**
   我读取你的 watchlist（或你直接给链接），挑一条做调研，写成报告提交。

## 报告格式

文件名：`YYYY-MM-DD-<topic-slug>.md`（如 `2026-07-09-vertical-farming.md`）。
结构化深度调研，建议七段：

1. **核心结论** — 先给一段可独立阅读的判断。
2. **这是什么 · 技术原理** — 概念定义 + 通俗类比。
3. **关键玩家与公司** — 主角、幸存者、反面教材。
4. **现状与进展** — 时间线复盘 + 当前走势，带数据。
5. **挑战与争议** — 命门、争议点、未解问题。
6. **未来时间线** — 近/中/远期观察坐标（标明是判断非预言）。
7. **延伸阅读** — 多源链接。

## 版权红线（同 `instruction.md` §2）

- 只写**原创综述** + **必附来源链接**；不复制原文段落，不绕过付费墙。
- 数据、结论尽量标注公开出处；分析性观点注明"基于公开资料"，非投资建议。

## 如何让我写一篇

- 在站点收藏若干条 → 对我说"从 watchlist 里挑 X 写一篇"；或
- 直接把新闻链接发我 → "从这条出发写未来技术调研报告"。
